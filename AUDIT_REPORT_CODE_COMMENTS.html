📖 Recommended Comment Guidelines

When to Comment

  • Why, Not What: Code shows WHAT it does; comments explain WHY
  • Complex Algorithms: Multi-step processes need walkthroughs
  • Non-Obvious Decisions: Business logic, security choices, performance tradeoffs
  • Gotchas & Edge Cases: Known issues, limitations, or assumptions
  • All Exported Functions: Should have JSDoc

JSDoc Template for Services

/** * Brief one-line description of what function does. * * Optional: Longer description explaining business context, * design decisions, or important behavior. * * @param {ParamType} paramName - What this parameter means * @param {object} config - Configuration object * @param {string} config.option - Option description * @returns {Promise<ReturnType>} Description of return value * @throws {AppError} CODE - When this error occurs and why * * @example * const result = await myFunction(input, { option: value }); * // Returns: { success: true, data: ... } */

Inline Comment Template for Complex Logic

// Why this approach: // [Explanation of design decision] // // Example scenario: // [Concrete example of what happens] // // TODO/FIXME if applicable: // [Known limitation or improvement needed] const result = complexLogic();

Security Comment Pattern

// SECURITY: [Threat being mitigated] // This prevents [attack type] by [mechanism] // See: [RFC/Doc reference if applicable] const sanitized = preventAttack(userInput);

🎯 Conclusion

The Navin backend codebase demonstrates selective excellence in documentation (Auth service is exemplary) but suffers from inconsistent coverage across modules. At 27% JSDoc coverage and sparse inline comments, the code fails to meet industry standards for maintainability and developer experience.

The primary gaps are:

  • Business Logic Not Explained: Why thresholds exist, how pagination works, why soft-deletes are used
  • Security Decisions Undocumented: Token revocation strategy, data normalization rationale
  • Complex Algorithms Lack Walkthroughs: Aggregation pipelines, anomaly detection, organization filtering
  • No Developer Onboarding Guides: Architects decisions trapped in implementer heads

  • Phase 2 (Weeks 3-4): 6-8 hours → Middleware + database patterns
  • Phase 3 (Ongoing): 2-3 hours → Enforcement & maintenance
